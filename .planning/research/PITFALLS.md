# Domain Pitfalls

**Domain:** PE CRM data model expansion — FastAPI + SQLAlchemy 2.0 + PostgreSQL + React (TanStack Query)
**Researched:** 2026-03-26
**Scope:** Adding ~40 columns to Contact/Company/Deal; 2 new sub-tables (DealCounterparty, DealFunding); 10 admin reference data tables; React UI for all new fields.

---

## Critical Pitfalls

Mistakes that cause migration failures, data corruption, or rewrites.

---

### Pitfall 1: Autogenerate Produces a Broken Migration Because the Initial Migration Uses `create_all`

**What goes wrong:** `alembic/versions/0001_initial.py` implements `upgrade()` by calling `models.Base.metadata.create_all(bind)`. This means Alembic's autogenerate has never recorded individual table definitions in its state. When you run `alembic revision --autogenerate`, Alembic compares the current model metadata against the revision chain — but because the chain contains only a raw `create_all`, autogenerate believes every table is new and generates `op.create_table(...)` for all of them instead of `op.add_column(...)` for new fields.

**Why it happens:** Alembic autogenerate works by diffing `target_metadata` against the reflected database schema. The `create_all` shortcut bypasses Alembic's own table-tracking entirely, so the migration chain has no memory of existing tables.

**Consequences:** Running the autogenerated migration against a live database that already has the tables from `create_all` will either fail with "table already exists" or — worse — silently skip already-existing tables while adding only new columns, depending on `--autogenerate` flags. The migration becomes non-reproducible.

**Prevention:**
1. Before writing any new models, run `alembic revision --autogenerate -m "snapshot_existing"` and inspect the output. If it produces `op.create_table` for tables that already exist, the migration chain is broken.
2. Fix by: run `alembic stamp head` on a database that already has the schema applied, then write the new migration manually with explicit `op.add_column(...)` calls. Do not rely on autogenerate for this milestone.
3. Write new migrations by hand: `alembic revision -m "add_pe_fields"` and use `op.add_column` / `op.create_table` explicitly.

**Detection:** Run `alembic revision --autogenerate --sql` (offline mode, no DB changes) and read the output before applying. If it contains `create_table` for Contact, Company, or Deal, stop and fix the chain first.

### Pitfall 2: `Base.metadata.create_all` in `main.py` Lifespan Silently Adds New Model Columns Outside Alembic

**What goes wrong:** `backend/api/main.py:43-44` calls `await conn.run_sync(Base.metadata.create_all)` during app startup. Any new column added to a SQLAlchemy model will be silently created by `create_all` in dev/test environments before any Alembic migration is written. The column exists in the database but Alembic has no record of it. The next `alembic revision --autogenerate` will not see it (because it already exists in the DB, matching the model), so no migration is generated. When the migration is deployed to a fresh environment, the column is absent.

**Why it happens:** `create_all` is idempotent on existing tables but additive — it adds missing columns only if the DBMS supports it, and on PostgreSQL it does not add columns to existing tables at all. However the combination of a broken initial migration plus `create_all` in lifespan means dev databases and production databases diverge silently.

**Consequences:** Production deploy fails with `column "xyz" of relation "deals" does not exist` on the first query after migration.

**Prevention:** Remove `create_all` from `lifespan` before writing any new models. Every schema change must go through an explicit Alembic migration. This is also flagged in CONCERNS.md but must be done before adding new columns or the schema drift will compound.

### Pitfall 3: All New Nullable Columns on Existing Tables Require `server_default` for Zero-Downtime Safety

**What goes wrong:** Adding a NOT NULL column to a table with existing rows without a `server_default` will fail with a PostgreSQL constraint violation. Even for nullable columns, if the application code path reads the new column without guarding against `None` and the column has not been backfilled, you get runtime `AttributeError` or serialisation failures.

**Why it happens:** PostgreSQL fills existing rows with `NULL` for new columns regardless of the Python-side `default=`. The Python `default=` on a SQLAlchemy `mapped_column` is an ORM-layer default applied only when inserting via SQLAlchemy — it is not a database-level default and does not backfill.

**Consequences specific to this expansion:**
- `Deal.revenue`, `Deal.ebitda`, `Deal.ev`, `Deal.equity_investment` — if added as NOT NULL with Python-only defaults, any existing deal row will cause a migration failure.
- Reference data FK columns (e.g., `Deal.transaction_type_id`) — if added as NOT NULL, migration fails on any existing deal.

**Prevention:**
- All new columns on existing tables must be nullable OR use `server_default=` with a literal value.
- Pattern for numeric fields: `mapped_column(Numeric(18, 2), nullable=True)` — never `nullable=False` without a `server_default`.
- Pattern for FK reference data: `nullable=True` always; enforce presence at the application layer, not the DB constraint, until all existing rows are backfilled.
- In the Alembic migration, use `op.add_column(table, sa.Column(..., nullable=True))` — never `nullable=False` on additive columns until a separate backfill migration has been run.

### Pitfall 4: Reference Data FK Values Hard-Deleted While Records Still Reference Them

**What goes wrong:** Admin deletes a Sector reference row. The Company rows that have `sector_id = <deleted-sector>` either fail FK constraints (if `ondelete="RESTRICT"`) or silently become orphaned (if `ondelete="SET NULL"`). If the frontend reads `company.sector.name` without guarding for `null`, the detail page crashes.

**Why it happens:** Reference data feels "safe to edit" because it is admin-managed. Teams assume that cleaning up old values is harmless. But FK relationships mean any deletion has downstream effects across all entities that ever used that value.

**Consequences:** Frontend crashes when rendering a Company whose sector was deleted. Broken list views if sector is used for grouping or filtering.

**Prevention:**
- Use `ondelete="SET NULL"` on all reference data FK columns (not RESTRICT or CASCADE).
- Add a `is_active` boolean to all reference data tables. Admin "deletes" set `is_active = False`, never physically delete. The row remains for historical display, but is excluded from dropdowns.
- Frontend: always render `sector?.name ?? "—"` — never `sector.name` without optional chaining.
- Seed the initial TWG values inside a migration, not in application startup code, so they survive database resets and CI environments.

### Pitfall 5: Amount Fields Stored Without Paired Currency Code

**What goes wrong:** Numeric amount fields (`revenue`, `ebitda`, `ev`, `equity_investment`, `aum`, `projected_commitment`, `actual_commitment`, `check_size`) are added to models and API schemas without a corresponding currency code column. Later, when currency support is needed (even for display purposes), the stored amounts have no currency context and are uninterpretable.

**Why it happens:** "Multi-currency conversion is out of scope" is interpreted as "currency codes are also out of scope." But storing amounts without currency means a USD 50M EBITDA and an SGD 50M EBITDA are indistinguishable in the database.

**Consequences:** Data must be re-collected from users when currency display is added. Import from the PE Blueprint template will fail because source data includes currency codes.

**Prevention:**
- Every amount field gets a paired `_currency` column: `Mapped[Optional[str]] = mapped_column(String(3), nullable=True)`.
- Default to `"USD"` as `server_default` for new deals, matching the existing `Deal.currency` pattern already in the model.
- For `DealFunding.projected_commitment` and `actual_commitment`: pair with a single `DealFunding.currency` column shared by both amounts on the same row (the funding commitment for one capital provider is always in one currency).
- For `DealCounterparty.aum` and `check_size`: pair each with its own `_currency` column since they come from different sources.
- Validated at the Pydantic schema level: if `amount` is not None, `amount_currency` must also not be None.

## Moderate Pitfalls

---

### Pitfall 6: TanStack Query Cache Serves Stale Reference Data After Admin Edits

**What goes wrong:** Reference data (Sectors, Transaction Types, etc.) is fetched once and cached indefinitely by TanStack Query. An admin adds a new Sector. The deal edit form still shows the old dropdown for all other open browser tabs — for the rest of the `staleTime` window.

**Why it happens:** Reference data is low-churn so `staleTime` is typically set high (or defaults to 0 but `gcTime` is 5 minutes). Mutations in `AdminPage` that add/edit reference rows do not know to invalidate query keys used in the deal/contact/company forms.

**Prevention:**
- Name reference data query keys consistently: `['ref', 'sectors']`, `['ref', 'transaction-types']`, etc.
- After any admin mutation that writes a reference table, call `queryClient.invalidateQueries({ queryKey: ['ref'] })` — this invalidates all reference data in one call using prefix matching.
- Set `staleTime: 5 * 60 * 1000` (5 minutes) for reference data — long enough to avoid re-fetching on every form open, short enough that admin changes are visible quickly.
- Do not use a single `['admin']` key for reference data — it will be hard to selectively invalidate later.

### Pitfall 7: React Edit Form With 40+ Fields Becomes Unmanageable as a Single Component

**What goes wrong:** The existing `ContactDetailPage` edit pattern maps over a list of field names and renders a plain `<Input>` for each. This works for 6 fields. With 15+ PE fields per entity — including dropdowns (reference data selects), date pickers, boolean toggles, multi-select arrays (board memberships, sector preferences) — a single flat form component becomes 400+ lines, unmaintainable, and rerender-expensive.

**Why it happens:** The incremental approach of adding fields one-by-one to an existing form does not force a structural decision until it is already a problem.

**Consequences:** Fields with different input types (select vs input vs date vs boolean) cannot all be handled by the `{fields}.map((field) => <Input>)` pattern. The form must be partially rewritten anyway to accommodate selects. If this is not planned upfront, the form is rewritten twice.

**Prevention:**
- Group fields into sections (Identity, Investment Preferences, Financial, Internal) and use `<Tabs>` or collapsible sections per group. The existing `ContactDetailPage` already uses `<Tabs>` for the read view — extend this to the edit view.
- Use `react-hook-form` (already in use in `ContactDetailPage`) with a field-config-driven approach: define a `FIELD_DEFS` array that includes `{ name, label, type: 'text' | 'select' | 'date' | 'boolean' | 'number' }` and render via a `<FieldRenderer>` component. This is more work upfront but scales to 40+ fields without per-field JSX.
- For reference data selects, create a single `<RefSelect refType="sectors" />` component that internally fetches `['ref', 'sectors']` from TanStack Query. This keeps form components decoupled from the fetch logic.

### Pitfall 8: DealCounterparty and DealFunding List Endpoints Trigger N+1 Queries

**What goes wrong:** `DealDetailPage` will need to display both a DealCounterparty list and a DealFunding list. If each sub-table row is fetched individually (one query per row to resolve the counterparty contact or company name), a deal with 20 counterparties issues 20 extra queries.

**Why it happens:** The existing `_base_deal_stmt` pattern does multi-table joins correctly (already joins Pipeline, Stage, User, Contact, Company). But new developers adding sub-table service methods default to loading related records via ORM relationship access (`counterparty.contact.full_name`) rather than writing an explicit join query.

**Consequences:** Deal detail page load time grows linearly with the number of counterparties. For a PE deal with 30+ investors tracked, this is noticeable.

**Prevention:**
- Write the `DealCounterpartyService.list_for_deal` method as a single query with explicit `.join()` to the Contact and Company tables, using `.label()` columns for display names — exactly as `_base_deal_stmt` does in `deals.py`.
- Never access relationship attributes on SQLAlchemy ORM objects from within an async service method without having explicitly loaded them via `selectinload()` or a join. Accessing `obj.relationship` in an async context triggers a lazy-load that raises `MissingGreenlet` in SQLAlchemy 2.0 async mode — it will not silently do N+1, it will raise an error. This is actually protective, but the error message is confusing on first encounter.
- Pattern: always use `select(...).join(...)` queries with labels in service methods. Reserve relationship access for places where `selectinload` has been explicitly declared.

### Pitfall 9: `ARRAY(Text())` Columns for Multi-Value Fields Are Filtered Incorrectly

**What goes wrong:** The existing `apply_tag_filter` in `_crm.py` uses `cast(column, Text).like('%tag%')` — already noted in CONCERNS.md as broken for exact tag matching. Any new multi-value fields (Contact sector preferences, board memberships stored as arrays) that use the same `StringList = MutableList.as_mutable(JSON()).with_variant(ARRAY(Text()), "postgresql")` type will inherit this bug if they use the same filter helper.

**Prevention:**
- For new array fields that need to be filterable, use the PostgreSQL `@>` containment operator: `Column.contains([value])` in SQLAlchemy, which generates `column @> ARRAY['value']`. This only works on the `ARRAY(Text())` dialect variant, not on the JSON variant for SQLite.
- Write a new helper `apply_array_filter(stmt, column, values)` that branches on dialect — PostgreSQL uses `@>`, SQLite uses `JSON_EACH` with exact match. Do not extend `apply_tag_filter`.
- For sector preferences specifically (Contact), decide up front whether it is stored as an array of FK UUIDs referencing the Sector table, or as a denormalized array of strings. Array of FKs is not directly supported by `ARRAY(UUID)` with FK constraints in SQLAlchemy easily — store as `ARRAY(Text())` containing UUID strings, or use a join table. Join table is cleaner if the sector list is admin-managed.

### Pitfall 10: Admin Reference Data Seeding Runs Every App Startup Instead of Once

**What goes wrong:** Reference data (pre-populated TWG values for Sectors, Transaction Types, etc.) is seeded in a startup hook or `create_all` call. Every time the app restarts, the seed logic runs. If the logic uses `INSERT ... ON CONFLICT DO NOTHING`, this is safe but wasteful. If it uses plain `INSERT`, it fails with duplicate key errors after the first run. If an admin has modified a seed value's label, the next restart overwrites it.

**Why it happens:** Seed data feels like "initial state" and developers put it in startup code alongside `create_all`.

**Consequences:** Admin edits to seed values are silently overwritten on next deploy. Or the app fails to start because of duplicate key violations.

**Prevention:**
- Seed all reference data inside an Alembic migration (the same migration that creates the reference tables). Use explicit `op.bulk_insert()` calls. This runs exactly once, when the migration is applied.
- Never seed reference data in application startup code.
- For updates to seed values in future milestones, write a new migration.

### Pitfall 11: SQLAlchemy 2.0 Async Lazy Load Raises `MissingGreenlet` on Relationship Access

**What goes wrong:** New service methods for DealCounterparty or DealFunding access a relationship attribute on an ORM object inside an `async` function without having explicitly loaded it. Example: `counterparty.contact` after a plain `select(DealCounterparty)` query. In SQLAlchemy 2.0 async mode, this raises `sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called; can't call await_only() here`.

**Why it happens:** Developers familiar with SQLAlchemy 1.x expect lazy loads to work. SQLAlchemy 2.0 async disables implicit lazy loads entirely.

**Consequences:** Runtime 500 errors on any endpoint that loads a sub-table and touches a relationship. Will manifest as soon as the first DealCounterparty detail view is loaded.

**Prevention:**
- Default to join-based queries with `.label()` columns for all display data (see Pitfall 8).
- When relationship loading is genuinely needed (e.g., loading a Deal with all its DealCounterparty rows), use `selectinload`: `select(Deal).options(selectinload(Deal.counterparties))`.
- Never use `.lazy="select"` (the ORM default) on new relationships in async context — always declare `lazy="raise"` on new relationship definitions to force a hard error at model declaration time rather than a confusing runtime error.

### Pitfall 12: Currency Code Validation Only on Frontend Leaves Invalid Codes in the Database

**What goes wrong:** Currency dropdowns in the React form are populated from the admin-managed Currencies reference table. But the API `PATCH /deals/:id` accepts a raw string for `currency` and `amount_currency` fields. A direct API call (or a bug in form submission) can store `"USD "` (trailing space), `"usd"` (lowercase), or a free-text value that is not in the Currencies table.

**Why it happens:** Frontend validation is visible and catches most cases. Backend validation is only added when a specific bug surfaces.

**Consequences:** Inconsistent currency codes mean `formatCurrency(value, currency)` in the frontend renders incorrectly or fails. Amount aggregations across deals in the same currency silently miss rows.

**Prevention:**
- The Pydantic schema for any field carrying a currency code must validate: `@field_validator('currency') def validate_currency(cls, v): return v.strip().upper()`.
- For amount+currency pairs: add a Pydantic model validator that asserts if `amount is not None` then `currency` must be a 3-character alphabetic string.
- Do not validate against the Currencies reference table at the Pydantic level (too tightly coupled) — but do validate at minimum that the code is 3 uppercase ASCII characters.

## Minor Pitfalls

---

### Pitfall 13: `DealDetailPage` Will Make 4+ Parallel Queries Per Sub-Table

**What goes wrong:** The existing deal detail page already notes 4+ parallel queries. Adding DealCounterparty and DealFunding panels adds 2 more TanStack Query calls per page load. Without pagination, a deal with 50 counterparties returns all 50 rows on page open.

**Prevention:** DealCounterparty and DealFunding list endpoints should default to `size=50` with pagination support from day one, even if the UI does not expose pagination controls initially. Enforcing the limit at the API layer prevents accidental full-table fetches.

---

### Pitfall 14: `position` / `sort_order` Column on Reference Data Tables Causes Position Drift

**What goes wrong:** If reference data tables have a `position` or `sort_order` column for admin-controlled ordering, updating a single row's position requires updating all subsequent rows' positions atomically. Non-atomic updates (update row A to position 3, then update row B to position 4) leave a window where two rows have the same position, violating any unique constraint.

**Prevention:** Use fractional positioning (float column, no uniqueness constraint) or a gap-based integer (positions in multiples of 100). For the initial milestone, alphabetical display ordering is simpler and sidesteps this entirely — add explicit sort_order only if the admin UI needs drag-and-drop reordering, which is out of scope.

---

### Pitfall 15: Legacy ID Column Collision in Contacts/Companies/Deals

**What goes wrong:** Adding a `legacy_id` column (for mapping to Intapp DealCloud records) as a unique constraint across the org will fail migration if any existing rows have non-null values that duplicate. More likely: the column is added as nullable without an index, making future lookups by legacy ID O(n) rather than O(log n).

**Prevention:** Add `legacy_id` as `nullable=True` with a composite unique constraint `UniqueConstraint("org_id", "legacy_id")` so the constraint is org-scoped. Add an index on `(org_id, legacy_id)`. In the migration, add the column as nullable with no constraint first, then add the constraint in a separate step (in case existing rows have nulls — PostgreSQL's unique constraint allows multiple NULLs without violation).

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|----------------|------------|
| Alembic migration for new columns | Autogenerate produces `create_table` instead of `add_column` due to broken initial migration | Use `alembic stamp head` + manual `op.add_column` migrations (Pitfall 1) |
| Removing `create_all` from lifespan | Breaks dev database setup if done without also writing a proper initial migration | Fix the initial migration first, then remove `create_all` (Pitfall 2) |
| Adding NOT NULL columns to existing tables | Migration fails on live data | All additive columns must be `nullable=True` or have `server_default` (Pitfall 3) |
| Reference data table creation + seed | Seed data re-applied on app restart | Seed in Alembic migration only, never in startup code (Pitfall 10) |
| DealCounterparty service implementation | `MissingGreenlet` on relationship access in async context | Default to join queries; use `lazy="raise"` on new relationships (Pitfall 11) |
| Admin reference data UI | TanStack Query cache serves stale dropdowns after admin edits | Consistent `['ref', ...]` query key prefix; invalidate on mutation (Pitfall 6) |
| Contact/Company/Deal edit forms | 40+ fields in a single flat component | Group into tabbed sections with a field-config-driven renderer (Pitfall 7) |
| Amount fields (revenue, EBITDA, EV, AUM, etc.) | Stored without currency context | Every amount column gets a paired `_currency` column (Pitfall 5) |
| Reference data FK relationships | Hard deletes break UI for existing records | Use `is_active` soft-delete on reference rows; `ondelete="SET NULL"` on FKs (Pitfall 4) |
